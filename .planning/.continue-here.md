- [ ] **CONSTRAINT: ALWAYS source `~/env_isaaclab/bin/activate` before launching Isaac Sim.** The aic-dt extension's rclpy parity publishers (`/joint_states`, `/tf`, `/tf_static`) require the Python 3.11 ROS Humble workspace lib dir to win LD ordering over `/opt/ros/humble/lib`. The lifecycle helper script (`isaacsim_launch.sh`) does NOT source activate — wrap any launch with `bash -c 'source ~/env_isaaclab/bin/activate && DISPLAY=${DISPLAY:-:0} bash ~/.claude/skills/isaac-sim-extension-dev/scripts/isaacsim_launch.sh launch aic-dt'`. Without this, `quick_start` fails with `UnsupportedTypeSupport: Could not import 'rosidl_typesupport_c' for package 'geometry_msgs'`. Verified empirically 2026-05-03.

- [ ] **CONSTRAINT: NEVER `mv DerivedDataCache aside; mkdir DerivedDataCache` for a "fresh test" without immediately running `prime_usd_cache.py restore` afterward.** This wiped 90 minutes of debug time on 2026-05-01. After ANY successful `quick_start`, run `prime_usd_cache.py snapshot`. The known-good backup at `~/.cache/ov/DerivedDataCache.bak.known-good` is the deterministic recovery command — `prime_usd_cache.py restore known-good`.

- [ ] **CONSTRAINT: Pre-existing assumptions in the project record can become stale without the record being updated.** Verified empirically 2026-05-03 with D-04 cable wedge: prior sessions concluded the cable wedges PhysX, but under current launch conditions (venv-activate + warm cache) the wedge no longer reproduces — empirical re-test showed sim_t advancing at 0.67x realtime with cable SetActive(True). When in doubt about a stale-feeling decision, **re-test before assuming the previous conclusion still holds**.

- [ ] **CONSTRAINT: Hot-reloading parity_publishers.py via `disable+enable extension` corrupts the running DigitalTwin instance.** `_ext_log_lines` not initialized in the second on_startup → every print() in any thread crashes → MCP server thread dies → must full-restart Isaac Sim. To iterate on parity_publishers.py code: kill + restart Isaac Sim (with venv-activate), don't try in-place hot-reload.

- [ ] **CONSTRAINT: ROS2 message type imports must come from the Python 3.11 workspace dist-packages, NOT from /opt/ros/humble (Python 3.10 layout).** Even with the 3.11 path on sys.path, Isaac Sim's startup-time bridge attempt caches partial rcl_interfaces / sensor_msgs imports pointing at the 3.10 path; subsequent `rclpy.create_node()` then fails with `UnsupportedTypeSupport`. The fix lives in `parity_publishers._force_python311_ros_paths()` and `_ensure_rclpy_clean_import()` — these MUST run before any rclpy import. Don't bypass them when adding new rclpy publishers/subscribers in Phase 2.

**Do not proceed until all boxes are checked.**
