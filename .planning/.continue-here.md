## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention |
|---------|-------------|----------|------------|
| MoveIt `ApplyPlanningScene` ADD-on-existing-id is **not idempotent** | First refresh succeeds; second silently fails (no log, no error). Spent multiple cycles before comparing against sim's working `_cmd_grasp_refresh` which uses wipe-then-push | **blocking** | Any scene-mutation flow that may run >1 time on same id MUST do REMOVE → 500ms wait → ADD. Mirror `_cmd_grasp_refresh` pattern. Already enforced in `_real_refresh_cups_thread` and `_real_refresh_legos_thread` |
| Empirical derivation > first-principles for marker frame axis conventions | First `R_marker_to_object` matrix was geometrically logical but produced cup tilted 96°. Empirical `R = R_marker_world.inv() * R_obj_known` got it right on first try | advisory | When marker frame axes are unclear, capture one good marker + sim ground-truth quat, derive matrix empirically. Document in `marker_geometry.py` |
| Stale in-memory config in long-running processes | Edited `bbox_catalog.json`, but `localize_yoloe` had already loaded the file at startup → stale catalog in memory → silent skips downstream | advisory | When editing config files consumed at process start, restart the process. Verify via `ros2 topic echo` of published config before assuming runtime state matches file |
| Threaded apply closures have buffered stdout — silence ≠ failure | Apply threads' success-log lines didn't print immediately; false-negative diagnosis | advisory | For MoveIt scene state verification: use `/get_planning_scene` service directly with `WORLD_OBJECT_NAMES \| WORLD_OBJECT_GEOMETRY` components. Don't rely on RViz visual or log lines |

<decisions_made>

- **marker→object transformation lives in aruco_camera_localizer config + function**, NOT control_gui or sim publisher — per user constraint "input to VLA package has to be the same in sim and real"
- **Function returns BOTH position offset AND orientation quaternion** — position-only fix would still leave cups visually wrong (lying on side instead of upright)
- **R_marker_to_object derived empirically** from `R_marker_world.inv() * R_object_known`, not from first-principles geometric reasoning — first geometric guess was wrong (ArUco PnP labels marker frame axes differently than I assumed)
- **No color→size inference anywhere** — vision detects color but not size; `_build_lego_geometry` falls back to SolidPrimitive.BOX with bbox catalog dims when size suffix missing
- **Schema is forward-compatible** — adding `box_top_marker`/`sphere_pole_marker`/etc. requires only function addition + config edit, no call-site changes
- **Y-sign in original config was wrong** (`Y=-0.005` should have been `Y=+0.005`) — comment claimed "-Y in marker frame = up on cup" but ArUco PnP outputs marker frame with Y up on cup pattern
- **Wipe-then-push for all refreshes** mirrors `_cmd_grasp_refresh` — `ApplyPlanningScene` ADD-on-existing-id is not idempotent
