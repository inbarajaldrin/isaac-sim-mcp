<objective>
Fix the user-hit bug where pressing Drop Refresh does not update RViz cup collision/visual markers when cup positions change in `/drop_poses`. Implement CONTEXT.md decisions D-05 through D-09.

**Root cause (diagnosed in session 695a7f93 turn 64):** `_cmd_drop_refresh` (L3081–L3089) clears `_drop_data`, then schedules `_add_cup_collision_objects` via `root.after(600)` and `_refresh_display_markers` via `root.after(700)`. The 600/700ms delays are shorter than the time between `/drop_poses` publishes in some cases, so the markers publish from empty data before the callback repopulates. Visual markers also depend on `_show_visual_var` toggle state, which may be off.

**Fix strategy:** Sequence against FRESH data using a `threading.Event`-gated pipeline on a background thread. No more `root.after` chain.
</objective>

<action>
Rewrite `_cmd_drop_refresh` (starting at L3081) to:

1. **Snapshot stale cups.** Capture the CURRENT set of cup names known to the GUI:
   ```python
   with self._drop_lock:
       stale_names = list(self._drop_data.keys())
   ```
   These are the cups whose collision objects and markers must be removed first.

2. **Remove stale collisions + markers.** Call `self._remove_cup_collision_objects()` (already exists at L2889) and publish DELETE MarkerArray for every id in `stale_names`. Reuse the DELETE-marker pattern from `_delete_visual_markers` (L2975–L2987) but iterate the stale_names list rather than hardcoding `range(3)`. Ensure the DELETE publishes before step 3.

3. **Invalidate `_drop_data` and arm a fresh-data event.** Under `self._drop_lock`, clear `_drop_data`. Set a new `self._drop_refresh_event = threading.Event()` (create the attribute in `__init__` if it does not already exist; initialize to `threading.Event()`). Modify `_drop_callback` at L2786 so that after successfully parsing a non-empty `/drop_poses` message, it calls `self._drop_refresh_event.set()` — but ONLY when the event is currently armed (not set). This avoids setting it on every incoming message.

4. **Wait for fresh data on a background thread.** Spawn a daemon thread that:
   ```python
   timeout_s = 2.0  # tune if flaky
   got_fresh = self._drop_refresh_event.wait(timeout=timeout_s)
   ```
   - If `got_fresh` is True: proceed to step 5.
   - If `got_fresh` is False: log `'Drop refresh: no /drop_poses message within {timeout_s}s — topic may be down'` with `warn` level and return from the thread. The stale cups stay removed (the user sees an empty scene, which is the correct signal).

5. **Atomically re-add collisions + markers from fresh `_drop_data`.** On the thread:
   - Call `self._add_cup_collision_objects()` (it already reads `_drop_data` under `_drop_lock` at L2849).
   - Directly call `self._publish_cup_visual_markers()` — NOT `_refresh_display_markers`. This bypasses the `_show_visual_var` toggle so the markers always reflect current truth on refresh (D-08). The `_show_visual_var` toggle still controls visibility at rest via `_toggle_visual_markers`; refresh is a user-requested truth push.
   - Schedule the GUI-thread listbox repopulate via `self.root.after(0, self._populate_drop_list)`.

6. **Remove the `root.after(500/600/700)` chain.** The new implementation runs entirely on the spawned thread with `_drop_refresh_event.wait` as the single synchronization point. No `root.after` with non-zero delays.

7. **Handle the "cup disappeared" edge case (D-09).** Because step 2 removed ALL stale collisions before step 4 waits for fresh data, a cup that disappears from `/drop_poses` is naturally dropped — step 5's `_add_cup_collision_objects` only iterates the NEW `_drop_data`. No stale entry survives.

8. **Do NOT alter** `_drop_callback`'s core parsing logic at L2786 beyond the single `self._drop_refresh_event.set()` call added under a `self._drop_refresh_event` truthy check.

9. Keep the method name `_cmd_drop_refresh` (auto-registered `~/drop_refresh` Trigger) — no rename.
</action>
