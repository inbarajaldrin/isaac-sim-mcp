- .planning/phases/07-control-gui-service-audit-ik-cleanup/07-CONTEXT.md §Decisions D-02
</read_first>
<action>
Add a method `_cmd_dump_services(self)` to the GUI class in `control_gui.py`. Place it alongside the existing auto-registered `_cmd_*` methods (anywhere after L3068's `_cmd_hot_reload_gui` is fine). Implementation:

1. Walk the tkinter widget tree starting from `self.root` using `winfo_children()` recursively. For every widget whose class matches `tk.Button` or `ttk.Button`, extract:
   - Widget text (via `cget('text')`)
   - The `command` callable attached to the widget (via `cget('command')` — tkinter returns the command as a name string registered in the interpreter; resolve to the Python callable via `self.root.tk.call('info', 'body', cmd_name)` OR use the command-registry pattern below).
   - The containing LabelFrame / tab title (walk `master.master.master...` until a `ttk.LabelFrame` or `ttk.Frame` with a `text` option is found)

2. Because tkinter's command-name lookup is awkward, add a parallel in-Python registry. In `_build_*_tab` methods where buttons are created, record every button registration in `self._button_registry: list[dict]` with keys `{'widget_text', 'command_repr', 'target_method_name', 'tab', 'section'}`. Populate this registry by either:
   - Wrapping `ttk.Button` / `tk.Button` constructors in a helper `self._register_button(parent, text, command, ...)` that logs the mapping and creates the widget, OR
   - Adding a post-init widget-tree walk that pulls `command=` from each button via `str(btn.cget('command'))` and matches back to a `_cmd_*` method when the callable's `__name__` starts with `_cmd_`.

   Use the helper-wrapper approach — it is deterministic and does not rely on tk command-name introspection. Replace every existing `tk.Button(...)` and `ttk.Button(...)` construction in `control_gui.py` with `self._register_button(parent, text=..., command=..., ...)`.

3. The `_cmd_dump_services` method returns a plain-text markdown table as the service response string (via `Trigger.Response.message`). Columns: `Tab | Section | Button | Bound Method | Registered Service | Mapping OK`. "Mapping OK" is `✓` when the bound callable resolves to a `_cmd_*` method present in `self._debug_services`, `WRAPPER` when it is a documented `_*_btn_*` thin wrapper that invokes a `_cmd_*` method, and `❌ INLINE LAMBDA` otherwise.

4. Also dump the list of registered `_cmd_*` services that have NO button binding (agent-only services, e.g. `~/drop_select`, `~/dump_services` itself). Format as a second table below the first.

5. Total response string length is allowed to exceed 1 KB — Trigger services have no size cap in practice for this use case.
</action>

<task id="01-02">
<title>Ban inline-logic lambdas on Button commands</title>
<read_first>
- ~/Projects/Exploring-VLAs/vla_SO-ARM101/src/so_arm101_control/so_arm101_control/control_gui.py (full file — every `command=lambda` site needs audit)
- .planning/phases/07-control-gui-service-audit-ik-cleanup/07-CONTEXT.md §Decisions D-01
</read_first>
<action>
For every `command=lambda` and `command=lambda: ...` in `control_gui.py`, classify the body:

- **Zero-logic forwarder** (e.g. `command=lambda: self._cmd_drop_refresh()`, `command=lambda: self._cmd_drop_point()`): ACCEPTABLE. These are explicit `_cmd_*` invocations with no inline branching, no inline computation, no multi-statement bodies. Keep them.

- **Thin UI-state wrapper** (e.g. `command=self._ik_btn_plan_execute` where `_ik_btn_plan_execute` reads `self._ik_valid` then calls `_cmd_plan_execute`): ACCEPTABLE when the wrapper method name follows the `_*_btn_*` pattern. Rename any existing ad-hoc wrappers to match this pattern.

- **Inline logic** (anything with inline arithmetic, variable capture of non-self state, multi-line `lambda: (a(), b())` tuples, `command=lambda e=evt: ...`): UNACCEPTABLE. Extract into a new `_cmd_*` method OR a `_*_btn_*` wrapper. The extracted method's action must be identical to the old lambda body.

For each extraction:
1. Name the new method `_cmd_<verb>_<noun>` when it represents user-facing functionality that should also be a debug service.
2. Name it `_<area>_btn_<verb>` when it is strictly a UI-state guard that forwards to an existing `_cmd_*`.
3. Update the Button `command=` to reference the method directly (no lambda wrapper) via `command=self._cmd_foo` — NOT `command=lambda: self._cmd_foo()`. Tkinter accepts bare callables.

Audit every `command=` in the file after the edit. The final state must match: every button's `command=` is either a bare method reference starting with `self._cmd_` or `self._<area>_btn_`. No `command=lambda` survives.
</action>

<action>
Create `~/Projects/Exploring-VLAs/vla_SO-ARM101/src/so_arm101_control/so_arm101_control/test_button_service_mapping.py`. The test must:

1. Import the control_gui module and its `ARM_JOINT_NAMES`, `JOINT_LIMITS` constants so the module is loaded without instantiating the ROS node.

2. Parse `control_gui.py` as source text (AST-based approach — do not instantiate the GUI). Using `ast`:
   - Find the `ControlGUI` class definition (or whatever the class is actually named — inspect the file first).
   - Collect all method names starting with `_cmd_` → set `CMD_METHODS`.
   - Collect all method names matching regex `^_[a-z_]+_btn_[a-z_]+$` → set `BTN_WRAPPER_METHODS`.
   - Walk every `ast.Call` node whose function is `self._register_button`. For each call, extract the value of the `command=` keyword argument. The value must be an `ast.Attribute` of the form `self.<name>` where `<name>` is in `CMD_METHODS ∪ BTN_WRAPPER_METHODS`. Otherwise fail the test with a clear message naming the button text and offending command expression.
   - Walk every `ast.Call` node whose function is `tk.Button` or `ttk.Button` DIRECTLY (i.e., any leftover untransformed button). Fail the test if any are found — all buttons must go through `_register_button`.
   - Walk every `ast.Lambda` inside a `command=` keyword anywhere in the file. Fail the test with a clear message — no `command=lambda` is permitted.

3. Name the primary test function `test_every_button_binds_to_cmd_or_wrapper`. Include at minimum these assertions (each with a descriptive `assert` message):
   - `assert not direct_button_calls, f'Direct tk.Button/ttk.Button calls must go through _register_button: {direct_button_calls}'`
   - `assert not inline_lambdas, f'Inline command=lambda bindings are forbidden: {inline_lambdas}'`
   - `assert all(cmd in CMD_METHODS | BTN_WRAPPER_METHODS for cmd in button_commands), f'Unmapped button commands: {button_commands - CMD_METHODS - BTN_WRAPPER_METHODS}'`

4. Add the test to `setup.py`'s `tests_require` list OR ensure it runs under `pytest` invoked from the package root. `colcon test --packages-select so_arm101_control` must execute it.

5. No network, no ROS node, no X11 — pure static analysis.
</action>
