# Phase 07.1: Widget Registry Expansion ‚Äî Full CLI-Level Control - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the button-only `_button_registry` built in Phase 7 Plan 07-01 into a full **widget registry** that covers every interactive tkinter widget in `control_gui.py` ‚Äî Spinboxes, Checkbuttons, Entries, Listboxes, Scales ‚Äî plus the Notebook tab switcher and read-only log Text widgets. Expose generic `~/list_widgets`, `~/get_widget_value`, `~/set_widget_value` Trigger services so agents can read and write any widget by name, not just click buttons.

**The "full CLI-level control" bar:** an agent should be able to type an IK target xyz+quat, toggle Ground Plane, select a drop listbox row by index, switch between FK/IK/Grasp/RViz tabs, and read the log output ‚Äî all via `ros2 service call` + `ros2 param set`. No screen coordinates, no image-recognition clicks, no user intervention.

**Target repo:** `~/Projects/Exploring-VLAs/vla_SO-ARM101` (single repo).
**Primary file:** `src/so_arm101_control/so_arm101_control/control_gui.py`.
**Sibling test file:** extend `src/so_arm101_control/so_arm101_control/test_button_service_mapping.py` OR create `test_widget_registry.py`.

**Why inserted mid-Phase-7:** Plans 07-03 (pixel-diff screenshot of FK/IK tabs) and 07-04 (IK smoke test ‚Äî type xyz+quat, click Plan & Execute) both require programmatic widget control. Landing 07.1 before them converts those plans' UAT from "user at keyboard" to "agent self-verifies."

</domain>

<decisions>
## Implementation Decisions

### Registration Pattern (D-01 through D-03)
- **D-01:** **Per-type factory helpers** mirror `_register_button` ‚Äî one helper per interactive widget type:
  - `_register_spinbox(parent, *, label, textvariable, from_, to, ...)` ‚Äî returns a live `tk.Spinbox` registered under `label` (+ optional `tab`, `section`).
  - `_register_check(parent, *, label, variable, ...)` ‚Äî returns a `tk.Checkbutton` (note: Checkbutton's `text=` IS the label; no separate `tk.Label`).
  - `_register_entry(parent, *, label, textvariable, ...)` ‚Äî returns a `tk.Entry`. `label` is read from the sibling `tk.Label` OR passed explicitly.
  - `_register_listbox(parent, *, label, ...)` ‚Äî returns a `tk.Listbox`. `label` taken from the enclosing `ttk.LabelFrame.cget('text')` or passed explicitly.
  - `_register_scale(parent, *, label, variable, from_, to, ...)` ‚Äî returns a `tk.Scale`.
- **D-02:** Every `_build_*_tab` method replaces direct constructors for these widget types with the factories. Pattern matches Plan 07-01's transformation of `tk.Button` ‚Üí `self._register_button`.
- **D-03:** The AST test (`test_button_service_mapping.py` extended, or a sibling `test_widget_registry.py`) enforces: direct `tk.Spinbox`, `tk.Checkbutton`, `tk.Entry`, `tk.Listbox`, `tk.Scale` (and their `ttk.` counterparts) constructions outside the factories cause the test to FAIL.

### Widget ID Naming (D-04 through D-07)
- **D-04:** **ID = the widget's `label` string, disambiguated by `@tab` suffix when duplicated.**
  - Example: `X`, `Y`, `Z` (IK target spinboxes), `Duration (s):@Grasp` vs `Duration (s):@Drop` (both exist).
  - `@tab` suffix is only added when the raw label collides with another widget's label elsewhere in the GUI. Implementation: during `_finalize_widget_registry` (called after `_setup_gui_thread` completes), scan for collisions and retroactively add `@tab` suffixes to all colliding IDs.
- **D-05:** For widgets where no natural label exists (Listboxes, Notebook), use the enclosing LabelFrame's text OR an explicit `label=` override passed by the developer.
- **D-06:** Notebook tab switcher gets a reserved ID: `tab`. `set_widget_value('tab', 'IK')` switches to the IK tab. `get_widget_value('tab')` returns the current tab name.
- **D-07:** Log Text widgets get IDs `process_log` and `error_log` (explicit, fixed). Setting them is rejected with `success=False, message='widget <id> is read-only'`.

### Service Signatures (D-08 through D-11)
- **D-08:** Declare two new ROS2 parameters in `__init__` (alongside the existing `ik_target`, `jaw_*_clearance_mm`, `tcp_clearance_mm`):
  ```python
  self.declare_parameter('widget_id', '')
  self.declare_parameter('widget_value', '')
  ```
- **D-09:** Three new manual services, registered in the existing `_srv_*` loop:
  - `~/list_widgets` ‚Äî no params. Returns tab-grouped markdown table: columns `Tab | Section | Label | Type | Current Value | Writable`. Also includes "Agent-only widgets" (no tab/section tags) and the read-only log entries.
  - `~/get_widget_value` ‚Äî reads `widget_id` param. Returns `success=true, message='id=<id>, type=<class>, value=<current_value>, extra=<type-specific state>'`.
  - `~/set_widget_value` ‚Äî reads `widget_id` and `widget_value` params. Coerces `widget_value` to the widget's expected type and writes. Returns `success=true, message='wrote <value> to <id>'` or `success=false` with a clear error.
- **D-10:** Agent-only widgets (no button binding) are listed in `list_widgets` output alongside buttons ‚Äî but `list_widgets` is a NEW table. The existing `dump_services` button table stays unchanged. Two tables coexist.
- **D-11:** `get_widget_value` / `set_widget_value` ALSO accept button IDs (the `_cmd_*` name) for reading button state (returns `enabled`/`text`/`visible`). Setting a button returns `success=false, message='buttons are click-only; use ros2 service call ~/<name>'`. This unifies the agent-facing API: one pair of services reads/writes every tagged interactive element, with per-type semantics.

### Listbox Semantics (D-12 through D-14)
- **D-12:** **Write: row index.** `set_widget_value` on a listbox parses `widget_value` as a non-negative integer. `listbox.selection_clear(0, END); listbox.selection_set(N)`. Invalid index returns `success=false, message='row N out of range (count=M)'`.
- **D-13:** **Read: full snapshot.** `get_widget_value` returns structured info in message:
  ```
  id=<id>, type=Listbox, value=<selected_text_or_empty>, extra=index=<N_or_-1>, count=<M>, items=[<row_0>|<row_1>|...]
  ```
  Items joined by `|` (choosing `|` over `,` because drop listbox entries contain comma-separated XYZ positions).
- **D-14:** Empty selection returns `value=, extra=index=-1, count=M, items=[...]`. Agents check `index=-1` to detect no-selection.

### Type Coercion Rules for `set_widget_value` (D-15 through D-19)
- **D-15:** `Spinbox` / `Scale`: coerce `widget_value` to float. If coercion fails ‚Üí `success=false, message='<id> expects float, got <raw>'`. If value outside the widget's `from_`/`to` range ‚Üí clamp silently and log a warning to the process log.
- **D-16:** `Checkbutton`: accept `'true'`, `'false'`, `'1'`, `'0'`, `'yes'`, `'no'` (case-insensitive). Coerce to bool and `variable.set(bool_value)`. Invalid ‚Üí error.
- **D-17:** `Entry`: value is used verbatim as string. `entry.delete(0, END); entry.insert(0, value)`.
- **D-18:** `Notebook` (tab switcher): value must match a tab text exactly (case-sensitive). `notebook.select(target_index)`. Invalid ‚Üí `success=false, message='tab <val> not found; available: FK, IK, Grasp, RViz'`.
- **D-19:** After any successful write to a widget backed by a `tk.Variable`, the write happens on the GUI thread (`self.root.after(0, ...)`). The service blocks until the write completes OR timeout (1.0s default) before returning `response`. This matches the pattern already used by `_srv_set_ik_target`.

### Scope of Registered Widgets (D-20 through D-22)
- **D-20:** **Interactive registration (write-capable):** `tk.Spinbox`, `ttk.Spinbox`, `tk.Checkbutton`, `tk.Entry`, `ttk.Entry`, `tk.Listbox`, `tk.Scale`, `ttk.Scale`, `ttk.Notebook` (for tab switching).
- **D-21:** **Read-only registration:** `tk.Text` widgets used as log output (`self._process_log`, `self._error_log`). `set_widget_value` rejects writes. `get_widget_value` returns the full Text contents (bounded to last 10 KB to prevent huge responses).
- **D-22:** **Explicitly NOT registered:** `tk.Label` (display-only, never interactive), `tk.Frame` / `ttk.Frame` / `ttk.LabelFrame` (containers), `tk.Scrollbar` (internal UI plumbing). Direct constructions for these are allowed (the AST test MUST NOT complain about them).

### Hot-Reload Handling (D-23)
- **D-23:** `_cmd_hot_reload_gui` currently tears down and rebuilds the widget tree. The widget registry must clear itself at the start of the rebuild (`self._widget_registry.clear()` in the hot-reload path before `_setup_gui_thread()` re-runs). Factory helpers repopulate the registry during the rebuild. Between the clear and the repopulate, `get_widget_value` returns `success=false, message='GUI rebuilding ‚Äî retry in 1s'`.

### Documentation and Audit (D-24)
- **D-24:** `scripts/regen_agent_debug_guide.sh` (created in Plan 07-01) extended to also call `~/list_widgets` and splice its output into `docs/AGENT_DEBUG_GUIDE.md` under a new `## Widgets` heading (alongside the existing `## Button ‚Üî Service Mapping`). Both tables are runtime-generated truth.

### Tab Auto-Switch on Set (D-26, added 2026-04-19)
- **D-26:** When `set_widget_value` targets a widget whose registry entry has a `tab` tag (FK/IK/Grasp/RViz), the service automatically switches the main notebook to that tab BEFORE performing the write. Purpose: agent-driven writes become immediately visible in the GUI without a separate `set_widget_value tab ...` call. Skipped for the notebook widget itself (setting `tab` IS the switch). Skipped for widgets without a tab tag (Ground Plane, log Text, etc.). The tab switch is dispatched via `root.after(0, ...)` so it happens on the GUI thread alongside the value write.

### End-to-End Verification (D-25)
- **D-25:** **Acceptance demo:** The plan's UAT must include an end-to-end agent-driven IK flow executed entirely via CLI (no GUI interaction):
  1. `ros2 param set /so_arm101_control_gui widget_id 'tab'`
  2. `ros2 param set /so_arm101_control_gui widget_value 'IK'`
  3. `ros2 service call /so_arm101_control_gui/set_widget_value std_srvs/srv/Trigger {}` (switches to IK tab)
  4. Set `widget_id='X'`, `widget_value='0.20'`, call `set_widget_value` (types 0.20 into X spinbox)
  5. Repeat for Y, Z, quaternion components
  6. Call `ros2 service call /so_arm101_control_gui/ik_plan_execute std_srvs/srv/Trigger {}` (plans + executes IK)
  7. Call `get_widget_value` with `widget_id='process_log'` to read the last log lines ‚Äî verify the expected "IK: elbow-up collision-free" message.
  8. Call `~/screenshot` to capture the result.
  
  This demo proves Success Criterion #6 (end-to-end agent-driven IK flow) from ROADMAP.md.

### Claude's Discretion
- Exact label-detection heuristic for Spinbox/Entry/Scale (walk the parent Frame looking for a sibling `tk.Label`, or require explicit `label=` kwarg). Recommend: prefer explicit `label=` kwarg to avoid layout-order fragility.
- Whether `list_widgets` returns markdown (matches `dump_services`) or JSON (cleaner for programmatic consumption). Recommend: markdown for human readability, with a `--json` variant if needed later.
- Internal structure of `self._widget_registry` dict (likely `{id: {type, tab, section, tk_var_ref, widget_ref, label, ...}}`, WeakRef for widget_ref to survive rebuilds).
- Timeout value for the GUI-thread write in `_srv_set_widget_value` (start 1.0s, tune if flaky).

- The user's framing: *"almost like a CLI ‚Äî the whole point of this so the agent gets full control of the app"* (session 249b0255). The "CLI" metaphor is the mental model: every interactive widget is a named argument.
- Inspiration: `~/.claude/skills/element-registry` skill ‚Äî especially the three operations (`list_elements`, `click_element`, `get_element_state`), the "dynamic discovery over static registration" principle, and the "ID by path, match by text" duality.
- Phase 7 Plan 07-01 Spark A (`get_widget_state`) already uses the `ik_target` param to accept a widget id. Phase 07.1 switches to dedicated `widget_id` / `widget_value` params for cleanliness ‚Äî but `get_widget_state` itself can stay on `ik_target` for backwards-compat (it was added before the rename proposal).
- The end-to-end demo in D-25 is NOT a theoretical target ‚Äî Plan 07-03's pixel-diff and Plan 07-04's IK smoke test directly depend on it working. Failing D-25 blocks those plans.
