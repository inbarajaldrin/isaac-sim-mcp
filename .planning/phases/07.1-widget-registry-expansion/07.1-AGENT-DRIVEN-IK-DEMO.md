## What this proves

An agent can now drive the full SO-ARM101 control GUI from a shell — no GUI clicks, no coordinate-based clicking, no screen reading required. Every interactive widget is addressable by a human-readable name (`X`, `Y`, `Ground Plane`, `Drop Topic`, etc.) and the write/read semantics are consistent across widget types. The CLI usability gotcha — `ros2 param set` auto-infers types — is documented: wrap values in single quotes to force string type (`widget_value "'0.20'"`).

The demo script is checked in and reproducible. Any future change that breaks widget control will break this script.

## Known CLI usability gotcha (worth documenting in AGENT_DEBUG_GUIDE.md)

`ros2 param set <node> widget_value 0.20` silently coerces `0.20` to a float parameter, then the service reads the old STRING value (which is still `'IK'` from the previous `widget_id` set or whatever was there). To force string type:

```bash
ros2 param set /so_arm101_control_gui widget_value "'0.20'"  # note outer double + inner single
```

Or equivalently, the `--param-type string` flag (ROS 2 Jazzy+). This quirk does not affect correctness — the service returns a clear error message when the type mismatch happens — but costs one trip in agent flows.
