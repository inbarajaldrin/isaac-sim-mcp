# BLOCKING CONSTRAINTS — Read Before Anything Else

> Each constraint below was discovered through actual failure this session.

- [ ] **CONSTRAINT: asyncio.run_until_complete inside MCP `execute_python_code` deadlocks Kit's main thread.** Manifestation: `await app.next_update_async()` hangs forever because the call IS the main thread, awaiting a future fulfilled by the same thread. All publishers stop emitting (their `on_frame` callbacks share the thread). Even SIGTERM/SIGINT are ignored — only SIGKILL recovers. **Mitigation**: never await frame settlement inside `execute_python_code`; set state and return, let Kit advance naturally. If you really need to wait for frames, do it from a separate process via the ROS topic. (Documented in CLAUDE.md "Gotchas" section.)

**Do not proceed until that box is checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| asyncio + execute_python_code deadlock | Same-thread await on Kit's main loop. Cost: 1 SIGKILL + lost in-session pose tweaks. | blocking | CLAUDE.md gotcha entry; review before any new `execute_python_code` block that uses asyncio. |
| Vision-LLM color matching | GPT/Claude eyeball-estimating diffuseColor drifts and disagrees. Wasted ~30 min per surface before switching tools. | advisory | Use `sim-color-matcher` skill (drag-box pixel sampling, deterministic ratio math). |
| Active-viewport publisher confusion | `/workspace_camera_sim` follows active main viewport, not the prim it was started against. Switching active camera silently changes what the topic publishes. | advisory | For deterministic capture: set active viewport first (without async wait), or use Replicator-backed `/wrist_camera_rgb_sim`. |
