## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Recording-while-driving via control_gui services | Arm doesn't move because services time out under DDS+GPU load | blocking | Use rosbag record-then-replay (option A) OR direct `/arm_controller/joint_trajectory` publisher (option B). Don't call control_gui services during recording. |
| Alpha-strip in render-thread callback | Drops RTF from 18 Hz to 1 Hz | blocking | Strip alpha at consumer side, never at publisher. The lerobot patch already in place handles RGBAâ†’RGB in cv2 inside `_msg_to_ndarray`. |

## Critical Anti-Patterns (do NOT repeat these)

Discovered through actual failure this session:

1. **ANTI-PATTERN: Calling `qs_refresh_all` + `qs_select` + `qs_play` while lerobot-record is running** â†’ Services time out, no lego selected, qs_play does nothing. **Mitigation**: validate pick-and-place works WITHOUT recording first (run `test_qs_cycle.sh` standalone), THEN add recording on top. 5. **ANTI-PATTERN: Multiple `ros2 launch control.launch.py` without proper teardown** â†’ Three concurrent control_gui processes claim the same node name. Always pkill+wait+verify before relaunch.

6. **ANTI-PATTERN: `set -u` in shell scripts that source ROS2 setup.bash** â†’ setup.bash references unset vars; script exits silently. Use only `set -o pipefail`.

7. **ANTI-PATTERN: Trusting microbenchmarks for render-thread perf** â†’ Microbench used cached fake data; production reads GPU-pinned memory which is much slower per access. Always test in production conditions for callback-perf claims.
